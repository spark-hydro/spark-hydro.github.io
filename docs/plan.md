# Conversion Plan: Turn the Current Portfolio-Style Codebase into a Full iHydro M&D Website

## 1. Goal

Convert the current Hugo + CareerCanvas codebase into a stable, maintainable **iHydro M&D research-group website** while preserving:

- Hugo as the site generator,
- Tailwind/PostCSS styling,
- dark mode and light mode,
- research/team pages already started,
- and a path for future lab expansion.

At the same time, remove or refactor the personal-portfolio assumptions that still control the homepage.

---

## 2. Guiding strategy

I recommend a **controlled refactor**, not a full rewrite.

Why:

- the repo already has the correct basic stack,
- the team page and research section structures are usable,
- the theme already supports modern responsive layout,
- and your requested changes mostly require better template control and content cleanup.

So the fastest safe path is:

1. stabilize the current theme behavior,
2. refactor homepage composition,
3. convert section naming and content to lab semantics,
4. remove random palette refresh behavior,
5. clean deployment and stale config.

---

## 3. Phase-by-phase plan

## Phase 0 — Verify deployment source before heavy edits

### Why this comes first

The public site appears different from the reviewed repo. Before doing a deep conversion, verify what is actually publishing `https://spark-hydro.github.io/`.

### Check

- Is GitHub Pages publishing from this repo?
- If yes, from which branch and folder?
- Is Vercel also being used?
- Is the public site currently generated by another Hugo/Hugo Blox repo?

### Deliverable

A confirmed deployment path such as:

- “This repo + main branch + GitHub Pages static publish,” or
- “This repo + Vercel only,” or
- “Another repo is live, this repo is staging.”

### Trade-off

Skipping this step risks spending time changing the wrong codebase.

### Current file

`themes/careercanvas/layouts/_default/index.html`

### Current behavior

Everything is hard-coded:

```go-html-template
{{ define "main" }}
{{ partial "hero.html" . }}
{{ partial "about.html" . }}

{{ partial "experience.html" . }}
{{ partial "technical.html" . }}

    {{ partial "about.html" . }}
  {{ end }}

  {{ if and $sections $sections.skills }}
    {{ partial "skills.html" . }}
  {{ end }}

  {{ if and $sections $sections.experience }}
    {{ partial "experience.html" . }}
  {{ end }}

  {{ if and $sections $sections.technical }}
    {{ partial "technical.html" . }}
  {{ end }}

  {{ if and $sections $sections.testimonials }}
    {{ partial "testimonials.html" . }}
  {{ end }}

  {{ if and $sections $sections.contact }}
    {{ partial "contact.html" . }}
  {{ end }}
{{ end }}
```

## Why this is the right first refactor

It solves your first request immediately without forcing a full homepage rewrite.

## Trade-off

This still keeps the old section files in the repo, but hidden. That is okay for the short term and safer than deleting them prematurely.

## Phase 2 — Make section labels content-driven instead of personal-template-driven

## Objective

Prepare the site for “About Us,” “Our Skills,” “Projects,” and “Let’s Connect” without fighting old i18n labels.

## Why needed

Some partials use page titles, but `about.html` and `contact.html` still use old i18n strings, which makes renaming inconsistent.

## Files to modify

- `themes/careercanvas/layouts/partials/about.html`
- `themes/careercanvas/layouts/partials/contact.html`
- possibly `i18n/*.toml` if you still want translation support for fallback labels

## 2.1 Refactor `about.html`

### Current issue

It starts with:

```go-html-template
## {{ i18n "aboutMe" }}
```

This locks the section to a personal “About Me” framing.

### Recommended replacement

```go-html-template
{{ with .Site.GetPage "about" }}
<section id="about" data-animate>
  <h2>{{ .Title }}</h2>
  {{ with .Params.intro }}{{ . | markdownify }}{{ end }}

  {{ if .Params.quickfacts }}
    <h3>{{ i18n "quickFacts" }}</h3>
    {{ range .Params.quickfacts }}
      <div>
        <h4>{{ .title }}</h4>
        {{ .value | markdownify }}
      </div>
    {{ end }}
  {{ end }}

  {{ with .Params.study }}{{ . | markdownify }}{{ end }}
  {{ with .Params.passion_title }}<h4>{{ . }}</h4>{{ end }}
  {{ with .Params.passion_text }}{{ . | markdownify }}{{ end }}
  {{ with .Params.mix }}{{ . | markdownify }}{{ end }}
  {{ with .Params.personal }}{{ . | markdownify }}{{ end }}
</section>
{{ end }}
```

### Then update content title

`content/en/about.md`

```yaml
---
title: "About Us"
...
---
```

## 2.2 Refactor `contact.html`

### Current issue

It uses contact-form labels as the section heading, which is not great if you want “Let’s Connect.”

### Recommended replacement

```go-html-template
{{ with .Site.GetPage "contact" }}
<section id="contact" data-animate>
  <h2>{{ .Title }}</h2>
  {{ with .Params.intro }}<p>{{ . }}</p>{{ end }}

  {{ if site.Params.formspreeendpoint }}
    <h3>{{ i18n "sendMeAMessage" }}</h3>
    <!-- existing form markup -->
  {{ end }}

  <h3>Contact Information</h3>
  {{ .Content }}
</section>
{{ end }}
```

### Then update content title

`content/en/contact.md`

```yaml
---
title: "Let’s Connect"
...
---
```

## Trade-off

This makes templates more flexible, but it means titles become content-managed rather than i18n-managed. That is usually the better fit for a lab website.

## Phase 3 — Freeze the palette and keep dark/light mode

## Objective

Stop the color changes on refresh while preserving dark and light themes.

## Files to modify

- `themes/careercanvas/layouts/_default/baseof.html`
- `themes/careercanvas/assets/js/dynamic-colors.js`
- optionally `config.toml`

## Best option

### Option A — Remove dynamic palette randomization entirely

This is the cleanest path.

#### In `baseof.html`
Remove the dynamic color script from the global bundle.

### Current logic includes

```go-html-template
{{ $colors := resources.Get "js/dynamic-colors.js" }}
```

and later minifies/fingerprints it and includes it.

### Replace with

Remove those lines completely.

If you want to keep the script available for future manual palette switching, wrap it behind a config flag:

```go-html-template
{{ if site.Params.enable_dynamic_colors }}
  {{ $colors := resources.Get "js/dynamic-colors.js" }}
  {{ if eq hugo.Environment "production" }}
    {{ $colors = $colors | minify | fingerprint }}
  {{ end }}
  <script src="{{ $colors.RelPermalink }}"></script>
{{ end }}
```

Then in `config.toml`:

```toml
[params]
  enable_dynamic_colors = false
```

### Result

- no random palette changes,
- dark mode still works,
- light mode still works,
- less visual drift.

## Optional improvement: fixed palette config

If you still want palette semantics but fixed color branding, add:

```toml
[params.brand_palette]
  primary = "#1E3A8A"
  secondary = "#3B82F6"
  accent = "#10B981"
```

and generate CSS variables once in `color-variables.html` instead of randomly in JS.

## Trade-off

- Removing JS randomization is simpler and more stable.
- Keeping palette data in config is more flexible for future theming.

I recommend: **remove the randomness now, move to config-based fixed variables later if needed**.

## Phase 8 — Decide what to do with the theme submodule

## Objective

Reduce long-term maintenance risk.

## Current issue

Your site depends on an external theme submodule repository.

## Two options

### Option A — Keep submodule

Good if:

- you want to track upstream theme improvements,
- and you are comfortable managing submodule pointers.

### Option B — Vendor the theme into the repo

Meaning:

- remove the submodule,
- copy the theme code directly into `themes/careercanvas/`,
- fully own it in the website repo.

### Recommendation

For a lab website with custom templates and ongoing branding changes, I recommend **vendoring the theme** unless you specifically need upstream theme sync.

## Trade-off

- **Keep submodule**: cleaner upstream separation, but more operational complexity.
- **Vendor theme**: easier day-to-day editing and deployment confidence, but you own all future maintenance.

## 11. Final recommendation

If I were converting this repo for long-term iHydro use, I would do it in this order:

1. verify deployment source,
2. add homepage section toggles,
3. disable dynamic color randomization,
4. clean config/site identity,
5. rewrite homepage content into lab language,

8. decide whether to vendor the theme.

That gives you a clean, stable site quickly without throwing away the current Hugo/Tailwind investment.
